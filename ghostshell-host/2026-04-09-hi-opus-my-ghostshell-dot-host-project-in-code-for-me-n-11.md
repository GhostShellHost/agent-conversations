## The Problem

SQLite's `AUTOINCREMENT` tracks the highest-ever ID in a `sqlite_sequence` table. Even if you delete records 45/46, the next insert would still be 47. You need to both move/preserve those records AND reset the sequence counter back to 28.

## Options

**Option A — Renumber 45 & 46 into an "archive" range + reset counter**
- UPDATE their `id` to something like `9045` / `9046` and `public_id` to `CENSUS-00009045` etc.
- UPDATE `sqlite_sequence SET seq=28 WHERE name='census_records'`
- Next real submission → ID 29, `CENSUS-00000029` ✓
- Records are preserved and still accessible by their new public IDs

**Option B — Flag them as archived, use a separate sequence table**
- Add a `display_id` column decoupled from the auto-increment
- More future-proof but requires a schema migration and code change in the worker

**Option C — Delete both + reset counter**
- If they're truly not worth keeping, cleanest option
- Same sequence reset as Option A

## Recommendation

**Option A** is the right call — it preserves the records (they're paid, they exist), gives them harmless "out of band" IDs that signal they're test/archived entries, and gets you back on track for record 29.

```sql
UPDATE census_records SET id=9045, public_id='CENSUS-00009045' WHERE id=45;
UPDATE census_records SET id=9046, public_id='CENSUS-00009046' WHERE id=46;
UPDATE sqlite_sequence SET seq=28 WHERE name='census_records';
```
